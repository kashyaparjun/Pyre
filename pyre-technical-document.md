Technical Architecture Document

BEAM-Powered Agent Framework for Python
Version 1.0  —  March 2026

## 1. System Overview
Pyre is a dual-runtime architecture. The orchestration layer runs on the BEAM virtual machine (via a pre-compiled Elixir release), and the execution layer runs on CPython. These two runtimes communicate over a Unix domain socket using a binary protocol. From the developer’s perspective, only the Python side is visible; the Elixir node is an implementation detail managed entirely by the library.
The fundamental design principle is separation of concerns: Elixir manages everything related to process lifecycle, concurrency, fault tolerance, and message routing. Python handles everything related to agent logic, LLM API calls, tool execution, and data processing. Neither runtime does the other’s job.

## 2. Bridge Protocol
The bridge is the most performance-critical component. It carries every message between the orchestration and execution layers. A slow or unreliable bridge would negate every benefit of the BEAM-powered architecture.
## 2.1 Transport Layer
Transport: Unix domain socket (UDS). On Windows: named pipe. We choose UDS over TCP because it eliminates network stack overhead, avoids Nagle’s algorithm buffering, requires no port allocation, and delivers sub-microsecond latency for small payloads on Linux. The socket is created in a platform-appropriate temporary directory at startup (/tmp/pyre-{pid}.sock on Unix).
## 2.2 Serialization
Format: MessagePack. We choose MessagePack over JSON because it is 2–5x faster at both encoding and decoding, produces smaller payloads (binary vs text), and supports binary data natively (important for state snapshots). Both Elixir (msgpax) and Python (msgpack-python) have mature, well-tested implementations.
We explicitly do not use Protocol Buffers or FlatBuffers because the schema management overhead is not justified for this use case. Agent state shapes change frequently during development; a schema-less format allows the developer to iterate without regenerating stubs.
## 2.3 Framing
Unix domain sockets are byte streams, not message streams. We need framing to delimit individual messages. Each message is prefixed with a 4-byte big-endian unsigned integer indicating the payload length:

[4 bytes: uint32 big-endian length][N bytes: MessagePack payload]

This is the same framing strategy used by Erlang’s distribution protocol. It supports payloads up to 4GB per message, which is far beyond any realistic agent state size. The implementation is trivial on both sides: write 4 bytes then the payload; read 4 bytes, allocate a buffer of that size, read until full.

## 2.5 Message Flow: Call Sequence
A synchronous call (equivalent to GenServer.call) follows this sequence:
- Developer calls agent.call("research", {"topic": "AI"}) in Python.
- The Python AgentRef serializes this as a bridge message with type execute and sends it to the Elixir node.
- The Elixir AgentServer GenServer receives the message in its handle_call callback.
- The GenServer packs the current agent state and the message into an execute envelope and sends it across the bridge to the Python worker.
- The Python worker looks up the handler for the agent_id in its registry, deserializes the state and message, and invokes the handler function.
- The handler runs the agent logic (including any LLM API calls, tool execution, etc.) and returns a result containing the new state and a reply value.
- The worker serializes the result into a result envelope and sends it back across the bridge.
- The Elixir GenServer receives the result, updates its internal state, and replies to the original caller.
- The Python AgentRef resolves the awaitable with the reply value.
The entire bridge round-trip (steps 3–8) adds approximately 0.2–0.5ms of overhead. For a typical agent invocation that includes an LLM API call taking 500–5000ms, this overhead is 0.01–0.1% of total latency.
## 2.6 Message Flow: Cast Sequence
An asynchronous cast (fire-and-forget, equivalent to GenServer.cast) follows the same path but the Elixir GenServer does not block waiting for a reply. The Python worker still executes the handler and returns the new state, but the original caller has already continued execution. The GenServer processes the state update in handle_cast asynchronously.

## 3. Elixir Core
## 3.1 Application Supervision Tree
The Elixir application starts with a top-level SystemSupervisor that manages all other components. The tree structure is:

SystemSupervisor (rest_for_one)
├── Bridge.Server        (the Unix socket listener)
├── Bridge.ConnectionPool (manages active connections)
├── NodeMonitor          (watches the Python process)
├── Registry             (Elixir Registry for agent name lookups)
└── AgentSupervisor      (DynamicSupervisor for user agents)
├── AgentServer:researcher_1
├── AgentServer:researcher_2
├── UserSupervisor:research_team (user-defined)
│   ├── AgentServer:fact_checker
│   └── AgentServer:summarizer
└── ...

The rest_for_one strategy at the top level is deliberate: if the Bridge crashes, everything downstream (NodeMonitor, Registry, AgentSupervisor) must restart because they depend on a functioning bridge. If only the AgentSupervisor crashes, the Bridge and Registry remain intact.
## 3.2 AgentServer GenServer
Each agent in the system is represented by an AgentServer GenServer. This is the heart of Pyre. The AgentServer is responsible for: holding the agent’s state as a serialized binary, managing the agent’s mailbox (which is the GenServer’s built-in mailbox), delegating computation to the Python worker via the bridge, and handling timeouts when the Python handler takes too long.
Key implementation details:
- State is opaque: The Elixir side never inspects or modifies agent state. It stores it as a binary blob (MessagePack-encoded) and passes it through to Python. This ensures zero coupling between the Elixir state representation and the Python state model.
- Timeout handling: Each call to the Python worker has a configurable timeout (default 30 seconds). If the handler exceeds this, the GenServer exits with {:timeout, :handler}. The supervisor restarts it with the initial state. The orphaned Python handler may eventually complete, but its result is discarded because no matching correlation_id exists.
- Crash propagation: When the Python handler returns an error envelope, the GenServer exits with {:handler_error, reason}. This triggers the supervisor’s restart logic. The key insight is that the crash happens in Elixir, not in Python, so the BEAM’s fault-tolerance machinery handles it.
## 3.3 NodeMonitor
The NodeMonitor process monitors the health of the Python worker process. It does this via two mechanisms: a periodic heartbeat (ping/pong over the bridge, every 5 seconds) and OS-level process monitoring (checking if the PID is still alive). If the Python process dies:
- NodeMonitor detects the failure (heartbeat timeout or OS signal).
- It notifies the SystemSupervisor.
- All pending bridge calls are resolved with {:error, :node_down}.
- AgentServers that were waiting for results crash and are restarted by their supervisors.
- NodeMonitor spawns a new Python worker process.
- The new worker re-registers all handlers (the handler registry is rebuilt from the Python-side handler map, which persists in the parent process).
- The system resumes normal operation.
The entire recovery sequence completes in under 2 seconds, most of which is Python process startup time.

## 4.1 Agent Base Class

The state must be a Pydantic BaseModel. This is a hard requirement, not a recommendation. Pydantic provides serialization via model_dump(), deserialization via model_validate(), and runtime type validation. This ensures that every state crossing the bridge is well-formed and serializable.
## 4.2 Worker Dispatch Loop
The Python worker runs an asyncio event loop that listens on the bridge socket and dispatches messages to the correct handler. The core loop is:
- Read 4 bytes from the socket (message length).
- Read N bytes (the MessagePack payload).
- Decode the envelope.
- Look up the handler for envelope.agent_id in the registry.
- Deserialize envelope.state into the agent’s Pydantic model.
- Deserialize envelope.message into a Python dict.
- Call the appropriate handler method (handle_call, handle_cast, etc.).
- Serialize the result (new state + reply) back to MessagePack.
- Write the response envelope to the socket.
The dispatch is fully asynchronous. Multiple handler invocations can be in-flight simultaneously (e.g., while one handler awaits an LLM API response, another handler can execute). This is where Python’s asyncio actually shines — I/O-bound concurrency within a single thread.
## 4.3 AgentContext
The AgentContext object is passed to every handler invocation. It provides the handler with the ability to communicate with other agents without breaking out of the process model:
- ctx.call(name, type, payload) — Synchronous call to another agent. Routes through Elixir’s registry and the target agent’s GenServer.
- ctx.cast(name, type, payload) — Asynchronous message to another agent.
- ctx.send_after(name, type, payload, delay_ms) — Schedule a message for future delivery. Implemented via Elixir’s Process.send_after.
- ctx.self — The current agent’s name/PID.
- ctx.spawn(AgentClass, name, args) — Spawn a new agent from within a handler.
Every operation on AgentContext goes through the bridge to Elixir. The handler never directly mutates another agent’s state or bypasses the mailbox. This preserves the isolation guarantees.
## 4.4 Handler Purity Constraint
Agent handlers must be functionally pure with respect to agent state. This means: the handler receives the current state and a message as arguments, and returns a new state. It must not store agent state in instance variables, class variables, global variables, or closures. Side effects (API calls, file I/O, database writes) are permitted and expected, but state must flow through the return value.
This constraint is what makes supervision work. When Elixir restarts a crashed agent, it creates a new GenServer with the initial state. The Python handler is stateless — it’s just a function waiting to be called. There is nothing to "restart" on the Python side. If the handler held state in instance variables, a restart would lose that state, creating subtle bugs.
The Pydantic model requirement enforces this at runtime: if a handler returns something that isn’t the declared state model, the serialization step fails with a clear error. Future versions may add a static analysis tool (via mypy plugin) to catch violations at type-check time.

## 5. Supervision Model
Pyre exposes the full OTP supervision model to Python developers. This section describes how Python supervision configuration maps to Elixir supervisor behavior.
## 5.1 Restart Strategies

Strategy
Behavior
Use Case
one_for_one
Only the crashed agent is restarted. Others are unaffected.
Independent agents: each researcher works on a different topic.
one_for_all
All agents under this supervisor are restarted.
Tightly coupled agents: if one fails, the others’ state is no longer valid.
rest_for_one
The crashed agent and all agents started after it are restarted.
Pipeline stages: if stage 2 crashes, stages 3 and 4 need to restart too.
## 5.2 Restart Intensity
Each supervisor has a max_restarts and within_ms configuration. If an agent exceeds max_restarts crashes within within_ms milliseconds, the supervisor gives up and terminates the agent permanently. This prevents restart loops where a fundamentally broken agent keeps crashing and restarting indefinitely, consuming resources.
Default values: max_restarts=3, within_ms=5000. These can be overridden per-agent in the supervision config.
## 5.3 Crash Propagation Path
When a Python handler throws an exception, the crash propagates through the system as follows:
- Python handler raises an exception (e.g., API timeout, validation error).
- The worker’s dispatch loop catches the exception and encodes it as an error envelope.
- The error envelope is sent across the bridge to the Elixir node.
- The AgentServer GenServer receives the error and calls {:stop, {:handler_error, reason}, state}.
- The OTP supervisor detects the GenServer exit.
- The supervisor applies the restart strategy (e.g., one_for_one: restart only this agent).
- A new AgentServer GenServer is spawned with the original initial state.
- The agent is back online and processing messages.
The developer’s handler code contains zero error handling for this path. The handler simply throws an exception. The supervision tree handles the recovery. This is the "let it crash" philosophy in practice.

## 6. State Management
Agent state in Pyre has a unique lifecycle: it is defined and typed in Python (as a Pydantic model), serialized and transmitted over the bridge, and stored in Elixir (inside the GenServer process). The state crosses the runtime boundary on every invocation.
## 6.1 State Serialization
The state round-trip on every call/cast invocation follows this path:
- Elixir GenServer holds state as a binary blob (MessagePack-encoded bytes).
- On invocation, the blob is sent to Python via the bridge.
- Python deserializes it via StateModel.model_validate(msgpack.unpackb(blob)).
- The handler runs, producing a new state (Pydantic model instance).
- The new state is serialized via msgpack.packb(new_state.model_dump()).
- The serialized blob is sent back to Elixir and stored in the GenServer.
This means agent state must be fully serializable by both Pydantic and MessagePack. Permitted types: primitives (str, int, float, bool, None), collections (list, dict, tuple), nested Pydantic models, datetime objects (serialized as ISO strings), enums, and bytes. Not permitted: functions, lambdas, class instances that aren’t Pydantic models, file handles, sockets, or any object with non-trivial __init__.
## 6.2 State Snapshots
Pyre supports system-wide state snapshots for crash recovery of long-running workflows. A snapshot captures the serialized state of every active agent, along with the supervision tree structure, agent metadata (names, supervision config), and agent group memberships.
Snapshots are triggered explicitly via system.snapshot() or automatically on a configurable interval. The snapshot is stored as a single MessagePack file on disk. Restoration via system.restore(path) re-creates the entire supervision tree and spawns all agents with their captured states.
Snapshots are not a substitute for application-level persistence (e.g., database writes). They are designed for resilience against process-level failures and machine restarts during long-running agent pipelines.

## 8.3 Throughput
The theoretical bridge throughput is 50,000–100,000 messages per second for small payloads, based on Unix domain socket benchmarks. In practice, the bottleneck for agent workloads is external API rate limits (typically 100–1,000 requests per minute for LLM providers). The bridge will never be the limiting factor for LLM-based agent workloads.
For computation-heavy agent pipelines that don’t involve external API calls, throughput can be increased by opening multiple bridge connections (one per CPU core) and having Elixir distribute work across them via round-robin.

## 10. Security Considerations
The Unix domain socket is protected by filesystem permissions (mode 0600, owner-only). No network-accessible ports are opened. The bridge protocol does not include authentication because both endpoints run on the same machine under the same user. If Pyre is ever extended to support distributed operation (Elixir on one machine, Python on another), TLS and mutual authentication would be required on the bridge.
Agent state is stored in memory only (in the GenServer process heap). It is not written to disk unless the developer explicitly calls snapshot(). Snapshot files should be treated as sensitive data and protected accordingly.
The bundled Elixir binary is verified via SHA-256 hash at startup. If the hash does not match the expected value (embedded in the Python package), Pyre.start() raises a SecurityError and refuses to boot.

## 11. Open Design Questions
- Streaming support: LLM responses often stream token-by-token. The current call/reply model does not support streaming. A new message type (stream_start, stream_chunk, stream_end) may be needed. The question is whether streaming state updates should be buffered and applied at stream_end, or applied incrementally per chunk.
- Multiple Python workers: For CPU-bound handlers, a pool of Python workers (one per core) would enable true parallelism. This complicates the bridge (routing, affinity, state consistency) but may be necessary for workloads involving local model inference or heavy data processing.
- State persistence: Should Pyre offer built-in persistent state (beyond snapshots)? ETS/DETS on the Elixir side could provide fast, durable key-value storage. But it adds complexity and potential consistency issues between persistent and in-memory state.
- Hot code reloading: Elixir supports hot code reloading natively. Could we expose this to Python developers? A handler update would involve re-registering a new handler function for an existing agent without restarting the agent or losing its state. This is technically feasible but UX design is non-trivial.
- Distributed mode: The BEAM natively supports clustering across machines. Could Pyre expose this so agents on machine A can communicate with agents on machine B? This would require replacing the Unix socket bridge with a network-aware transport and adding service discovery. It is a natural extension but a significant engineering effort.
